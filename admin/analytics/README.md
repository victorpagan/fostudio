# FO Studio Analytics Pipeline (v1)

This workspace generates analytics artifacts for the admin dashboard and weekly business reporting.

## Directories

- `inputs/`: CSV fallback/mock inputs.
- `normalized/`: normalized machine-friendly datasets generated by scripts.
- `outputs/`: dashboard/report artifacts (`metrics.json`, `trends.json`, `alerts.json`, `weekly-report.md`, `weekly-report.json`).
- `scripts/`: TypeScript ingestion/compute/report scripts.
- `config/`: metric and alert definitions.

## Assumptions (v1)

- Timezone: `America/Los_Angeles`
- Week definition: ISO week (Monday start)
- Ingestion mode: Supabase first, CSV fallback
- Weekly utilization denominator: `168` hours (`config/metric-definitions.ts`)
- Revenue includes membership payments, guest booking-linked revenue, credit topups, and hold topups
- Email integration is advisory only (recommendations generated, nothing auto-sent)

## Environment for live ingestion

Set these to enable live Supabase ingestion:

- `SUPABASE_URL`
- `SUPABASE_SERVICE_KEY` (preferred) or `SUPABASE_KEY`

If these are missing or live queries fail, scripts use CSV files in `inputs/`.

## API access for chat/integrations

Use the analytics export endpoint:

- `GET /api/internal/analytics/outputs`
- `POST /api/internal/analytics/run` (runs pipeline, then returns latest outputs metadata)

Auth options:

- `Authorization: Bearer <ANALYTICS_EXPORT_SHARED_KEY>`
- `x-analytics-key: <ANALYTICS_EXPORT_SHARED_KEY>`
- admin session (same as admin dashboard APIs)

The shared key is read from the secrets store key:

- `ANALYTICS_EXPORT_SHARED_KEY`
- Optional dedicated run key: `ANALYTICS_RUN_SHARED_KEY`

Example:

```bash
curl -sS "$APP_URL/api/internal/analytics/outputs" \
  -H "Authorization: Bearer $ANALYTICS_EXPORT_SHARED_KEY"
```

```bash
curl -sS -X POST "$APP_URL/api/internal/analytics/run" \
  -H "Authorization: Bearer $ANALYTICS_EXPORT_SHARED_KEY" \
  -H "Content-Type: application/json" \
  -d '{"requireSupabase": true}'
```

## Remote run/fetch via fo.studio

Use these commands when running analytics through deployed endpoints (instead of local pipeline execution):

```bash
pnpm analytics:remote:run
pnpm analytics:remote:outputs
```

Defaults:

- Base URL: `https://fo.studio`
- Auth: values loaded from repo `.env` (`ANALYTICS_RUN_SHARED_KEY` and/or `ANALYTICS_EXPORT_SHARED_KEY`)
- Run mode sends `{ "requireSupabase": true }` so ingest fails rather than falling back to CSV.

Options:

```bash
pnpm analytics:remote:run -- --allow-fallback      # send requireSupabase=false
pnpm analytics:remote:run -- --url=https://fo.studio --json
pnpm analytics:remote:outputs -- --url=https://fo.studio --json
```

You can also set `ANALYTICS_APP_URL` if you want a different default host.

Response includes:

- `metrics`
- `trends`
- `alerts`
- `weeklyReportMd`
- `weeklyReportJson`
- `generatedAt`
- `freshness`
- `missingFiles`
- `authMode`
- `storage` (`supabase` or `filesystem`)
- `source`

To force Supabase and fail (instead of fallback):

```bash
ANALYTICS_REQUIRE_SUPABASE=1 pnpm analytics:ingest
```

You can also run each script with:

```bash
pnpm exec tsx admin/analytics/scripts/ingest_memberships.ts --require-supabase
pnpm exec tsx admin/analytics/scripts/ingest_bookings.ts --require-supabase
```

## Commands

Run from repo root:

```bash
pnpm analytics:ingest
pnpm analytics:compute
pnpm analytics:report
pnpm analytics:publish
pnpm analytics:all
```

Optional target week:

```bash
pnpm exec tsx admin/analytics/scripts/compute_metrics.ts --week-of=2026-04-06
```

`--week-of` expects an ISO date and is interpreted in `America/Los_Angeles`.

Verify actual ingest source from manifests:

```bash
cat admin/analytics/normalized/memberships.raw.json | jq '.manifest.source'
cat admin/analytics/normalized/bookings.raw.json | jq '.manifest.source'
cat admin/analytics/normalized/revenue.raw.json | jq '.manifest.source'
```

## Input schemas (CSV fallback)

### `inputs/memberships.csv`

Required columns:

- `membership_id`
- `user_id`
- `customer_id`
- `tier` (`creator`, `pro`, `studio_plus`)
- `status` (`active`, `past_due`, `canceled`, etc.)
- `amount` (USD)
- `created_at` (ISO date/datetime)

Optional columns:

- `cadence`
- `canceled_at`
- `current_period_start`
- `current_period_end`
- `last_paid_at`

### `inputs/bookings.csv`

Required columns:

- `booking_id`
- `customer_id`
- `start_time` (ISO)
- `end_time` (ISO)
- `status`
- `hours`
- `revenue` (USD)
- `booking_type` (`member`/`guest`)
- `channel`

Optional columns:

- `user_id`

### `inputs/revenue.csv`

Required columns:

- `date` (ISO)
- `source` (`membership`, `guest_booking`, `credit_topup`, `hold_topup`)
- `amount` (USD)

Optional columns:

- `user_id`
- `customer_id`
- `order_id`
- `tier`
- `cadence`

### `inputs/ads_meta.csv` and `inputs/ads_google.csv`

Required columns:

- `date` (ISO date)
- `campaign`
- `spend` (USD)
- `clicks`
- `impressions`
- `conversions`
